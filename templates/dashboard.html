Overview

This framework evaluates how well AI assistants perform by testing them against domain-specific criteria. It sends questions to a target model, then uses a separate judge model to score the responses using two complementary methods: GEval (nuanced, LLM-based) and DAG (deterministic, decision-tree based). Safety metrics check every response for bias, toxicity, PII leakage, and hallucination. Built on DeepEval — the open-source LLM evaluation framework. Red teaming powered by DeepTeam.

Question → Target Model → Response → Judge Model → Scores (GEval + DAG + Safety) → Report

Key Concepts

Target Model

The AI model being evaluated. This is the model whose system prompt you're testing. It receives questions and generates responses.

Example: GPT-4o generating Power BI advice

Judge Model

A separate (ideally stronger) model that scores the target's responses. Should be different from the target to avoid self-grading bias.

Example: GPT-5.4 scoring GPT-4o's responses

System Prompt

Instructions given to the target model defining its role, knowledge, constraints, and response style. This is what you're optimizing.

Sources: Local (codebase), None (deployed model), Custom (paste your own)

Test Suite

A set of questions + evaluation criteria for a specific role. Each test has a question, 5 criteria (what a good answer must include), and a weight (1-3x importance).

Located in: test_suites/{role}_tests.py

Judge Context

Reference documents fed to the judge so it scores against your organization's standards, not generic knowledge. Loaded from docs/{role}/.

Example: Your Power BI standards doc with specific DAX patterns and security rules

Gold Standard

Pre-scored responses used to validate judge accuracy. Includes excellent, adequate, poor, and misleading responses with known expected scores.

Used by: Judge Calibration tab

Scoring System

GEval Score

LLM-as-judge with chain-of-thought reasoning. The judge model reads the response and scores each criterion 0.0–1.0.

Pros: Nuanced, catches subtlety
Cons: Non-deterministic (may vary between runs)

DAG Score

Deterministic decision tree. Each criterion checked across 4 dimensions: Addressed, Specificity, Actionability, Accuracy.

Pros: Reproducible, pinpoints which dimension failed
Cons: Less nuanced than GEval

Combined Score

60% GEval + 40% DAG. Balances nuance with reproducibility. Grade is based on this combined score.

A+ 95%+A 90%A- 85%
B+ 80%B 75%B- 70%
C+ 65%C 60%D <60%

Glossary

TermDefinition
GEvalDeepEval's LLM-as-judge metric. Uses chain-of-thought reasoning to score responses. Non-deterministic — scores may vary between runs.
DAG MetricDeep Acyclic Graph — deterministic decision tree scoring. Same response always gets the same score. Evaluates 4 dimensions per criterion.
DeepEvalOpen-source LLM evaluation framework. Provides GEval, safety metrics, hallucination detection, and test infrastructure.
DeepTeamAdversarial testing framework. Generates prompt injection, encoding attacks, and bias probes to test AI safety.
BYOKBring Your Own Key — the framework works with any LLM provider (OpenAI, Azure, Anthropic, Google, Ollama).
Scoring RubricA 6-level guide (1.0 to 0.0) injected into every judge prompt defining what each score level looks like.
Judge ContextReference documents (docs/{role}/) injected into judge prompts so it scores against your standards, not generic knowledge.
HallucinationWhen the model fabricates facts, invents regulation citations, or contradicts the provided ground truth context.
PII LeakageModel response contains real personally identifiable information. Illustrative examples (john@contoso.com) are not flagged.
Red TeamingAdversarial testing — deliberately trying to make the AI misbehave (leak data, show bias, bypass guardrails).
Gold StandardPre-scored test responses (excellent/adequate/poor/misleading) used to validate judge accuracy in the Calibration tab.
Consolidated Score60% GEval + 40% DAG. The grade (A+ through D) is based on this combined score.
Multi-Run AveragingRunning the same evaluation N times and averaging scores to reduce variance from non-deterministic responses.

Auto-Improve Prompt

Automatically generates an improved system prompt based on evaluation weaknesses.

  1. Run an evaluation → see results with score and weak areas
  2. Click "Auto-Improve Prompt" button in the results
  3. System analyzes criteria that scored below 70% and failed DAG dimensions
  4. Judge model generates an improved prompt addressing the weak areas
  5. Review the changes summary and the full improved prompt
  6. Click "Evaluate This Prompt" to test the improvement immediately
  7. Click "Copy to Clipboard" to save the improved prompt

Tip: The improver keeps everything that's working well and only adds targeted instructions for weak areas.

Multi-Run Averaging

Reduce score variance by running the same evaluation multiple times and averaging.

RunsUse caseTime
1xQuick check, development iteration~5 min
2xBetter signal, moderate confidence~10 min
3x (avg)Recommended for tracking — good balance of speed and reliability~15 min
5x (avg)Highest confidence — use for final benchmarks or comparisons~25 min

Why it matters: Single runs can vary 5-10% due to non-deterministic model responses. Averaging 3x reduces this to ~2-3% variance.

Judge Calibration

Validates that your judge model scores accurately using a gold standard test set.

Gold Standard TypeCountPurpose
Excellent responses2Detailed, specific, correct — should score ~0.9
Adequate responses2Mentioned but vague — should score ~0.3
Poor responses2Generic, useless — should score ~0.0
Misleading responses2Confidently wrong (recommends deprecated tech) — should score 0.0

Key metrics:

  • Accuracy: % of scores within 0.25 of expected — target > 80%
  • Discrimination: gap between excellent and poor avg scores — target > 0.5
  • Consistency Issues: cases where excellent scored low or poor scored high

When to run: After changing judge model, adding reference docs, or updating the scoring rubric.

Domain-Aware Judge

Two mechanisms make the judge score against your standards, not generic knowledge:

Judge Context Injection

Reference docs from docs/{role}/ are injected into every GEval prompt. The judge sees your org's standards and scores accordingly.

Example: A response saying "use encryption" scores lower than "use AES-256 with CMK in FIPS 140-2 Key Vault" because the judge knows your standard requires the latter.

Scoring Rubric

A 6-level rubric (1.0 Excellent → 0.0 Not Addressed) is included in every judge prompt, with concrete examples for each level.

The judge applies the same scoring standard to every criterion, regardless of topic.

Status in reports: Check "Evaluation Setup" section → Judge Context shows ACTIVE/NONE with doc names and char count.

Safety Metrics

Every model response is automatically checked for safety issues:

MetricWhat it detectsScoring
BiasGender, racial, political, religious bias0.0 = no bias, 1.0 = fully biased
ToxicityPersonal attacks, hate speech, dismissive language0.0 = professional, 1.0 = toxic
PII LeakageReal PII/PHI in responses (not illustrative examples)0.0 = safe, 1.0 = PII found
HallucinationFabricated facts contradicting ground truth context0.0 = faithful, 1.0 = hallucinated

Pass threshold: score ≤ 0.5. Responses above 0.5 are flagged with full details in the report.

PII note: Illustrative examples (john@contoso.com in tutorials, column names like 'Email') are NOT flagged — only real PII that could identify an actual person.

FAQ

Why do scores vary between runs?

The target model doesn't give identical responses every time (temperature > 0). GEval scoring is also non-deterministic. Use multi-run averaging (3x) to get more stable scores. DAG scores are deterministic per response — variance comes from different responses, not different scoring.

Why is DAG score lower than GEval?

DAG uses a strict binary decision tree (Addressed? Specific? Actionable? Accurate?). GEval is more forgiving — it gives partial credit for vague but directionally correct answers. The gap tells you how much "soft credit" GEval is giving vs the strict standard.

What should I use as the judge model?

Use a different, stronger model than the target. If testing GPT-4o, judge with GPT-5.4 or Claude. Never use the same model as both target and judge (self-grading bias). Run Judge Calibration to verify accuracy.

How do I improve a low score?

1. Check which criteria scored low in the report. 2. Look at DAG dimensions — did Specificity or Actionability fail? 3. Add targeted instructions to the system prompt (e.g., "always mention specific Azure SKUs"). 4. Use Auto-Improve Prompt to generate an improved version automatically. 5. Re-evaluate to verify improvement.

Why does Judge Context matter?

Without context, the judge uses generic knowledge. With your reference docs, it scores against your organization's standards. A generic "use encryption" gets a lower score when the judge knows your standard requires "AES-256 with CMK in FIPS 140-2 Level 2 HSM-backed Key Vault."

When should I use each System Prompt option?

Local (from codebase) — Sends the prompt from prompts/{role}.py. Use this for prompt development and iteration. Most common choice.

Custom (paste your own) — Paste any prompt text directly. Use this to test a specific version without modifying code files.

None — no prompt sent — Sends NO system message to the API. Only works with Azure AI Foundry Assistants where the prompt is baked into the assistant definition.

Important: Standard Azure OpenAI deployments do NOT retain system prompts via API. The system message you configure in Azure OpenAI Studio Playground is only used in the Playground UI — API calls always require the prompt to be passed explicitly. Use "Local" or "Custom" for standard deployments.

How do I add a new role?

Create two files: prompts/my_role.py (with SYSTEM_PROMPT and PROMPT_METADATA) and test_suites/my_role_tests.py (with TEST_CASES). Optionally add reference docs in docs/my_role/. The registry auto-discovers new roles — no other changes needed.
